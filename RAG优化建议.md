# RAG 优化建议

本文档基于当前仓库实现整理：`backend/app/tools/rag_tool.py` 中的 `query_chroma`（Chroma + OpenAI 兼容 Embeddings）、`backend/app/agents/rag_agent.py` 中的 `RAGTool`（检索工具节点，证据交给 Summarizer）。优化可按 **数据与分块（入库侧）**、**检索**、**编排**、**工程**、**评估**、**选型** 分层推进。**评估与选型均以文档约定与人工/脚本执行为主**，不绑定仓库内具体实现代码。

---

## 1. 检索侧

### 1.1 查询改写 / 多查询

当前检索查询一般为用户子任务原文直接 embedding。对口语、多轮指代、过短或冗长输入，可增加一层：

- **查询规范化**：补全省略的主语/业务实体（如规则名、产品类目）。
- **HyDE**：让模型先生成「假设答案」再对假设答案做向量检索（适合知识库文风与问题文风差异大的场景）。
- **子查询拆分**：复合问题拆成多条检索，再合并去重与重排。

### 1.2 混合检索

纯向量对专有名词、型号、条款编号、固定短语往往不稳。在具备全文或关键词索引的前提下，可做 **BM25/关键词 + 向量**，分数融合（如 RRF），再进入重排或直接进入上下文。

### 1.3 重排序（Rerank）

流程建议：Chroma 先取较大 `n_results`（如 15～30），再用 **cross-encoder 或专用 rerank API** 压缩到 3～5 条。通常比单纯调大/调小 `top_k` 更省 token 且相关性更好。

### 1.4 相似度阈值与无结果策略

避免「召回到条但全不相关」：结合 Chroma 返回的 `distance`（需与集合的 `hnsw:space` 及实现约定一致，区分 distance 与 similarity）设置信阈值；低于阈值时走澄清、拒绝或扩大检索，减少下游被噪声带偏。

### 1.5 去重与多样性（MMR）

同一文档多个相邻 chunk 容易占满 top-k。可按 `source` / `doc_id` 做 **每文档配额**，或使用 MMR 在相关性与多样性之间折中，使上下文覆盖不同段落。

## 2. 数据与分块（在 AICODE / addrag 侧，但影响本服务）

本服务的 RAG 路径是：**addrag 写入 Chroma** → **本服务按相同 embedding 模型与集合配置查询**。因此「数据长什么样、怎么切块、元数据是否完整」决定了检索上限；代码里能做的多为阈值、重排、过滤，**无法替代劣质分块**。

### 2.1 职责边界：addrag 与本服务各自保证什么

| 环节 | 建议由 AICODE / addrag 保证 | 本服务（Agents）侧可依赖的契约 |
|------|-----------------------------|--------------------------------|
| 向量空间 | 建库时 `hnsw:space` 与所选距离一致（当前与 cosine 对齐，见 `rag_tool._assert_addrag_cosine_space`） | 启动或查询前校验集合 metadata，避免错库错空间静默劣化 |
| 向量维度与模型 | 入库与查询使用 **同一 embedding 模型、同一 API 版本行为** | `RAG_EMBEDDING_*` 与 AICODE 配置一致；换模型必须全量重嵌入 |
| 文档内容与切分 | 原文质量、去重、编码、清洗（HTML/表格/页眉页脚） | 消费 `documents` 与 `metadatas` 字段；不在本服务「修复」坏 HTML |
| 分块与重叠 | chunk 边界、overlap、父子块策略 | 通过 `snippet` 长度与 trace 观察是否截断严重 |
| 元数据 | `source`、`chunk_id` 等与业务一致的 schema | `query_chroma` 已读取 `source`/`doc`、`chunk_id`、`ids` 等；可扩展 `where` 过滤 |

**原则**：凡影响向量几何与倒排邻域的决策，尽量在 **入库流水线** 固定下来并版本化；本服务只做 **查询时** 的轻量策略（阈值、rerank、元数据过滤）。

### 2.2 分块粒度：过大与过小的权衡

- **块过大**  
  - 优点：单块上下文完整，适合「整段政策说明」。  
  - 缺点：embedding 是整段压缩向量，用户只问其中一句时，**整段语义被平均**，召回排名下降；送入 LLM 时 token 浪费、噪声多。

- **块过小**  
  - 优点：向量更「尖锐」，匹配具体问题更准。  
  - 缺点：丢失跨句依赖（「上述情形除外……」指代上文）；相邻法条被切断。

**实践建议**：以 **语义完整单元** 为首选切分边界（小节标题下的一段、一条规则、一个 FAQ 问答对），再在该单元内用 **目标 token 数**（如 256～512 中文字符等价量级，需按 embedding 模型上下文与成本微调）做二次切分；必要时对长条使用 **带重叠的滑动窗口**（见 2.4）。

### 2.3 结构化切分（优于「固定字数一刀切」）

在 addrag 中优先利用文档结构，减少「半句话 + 半张表」的块：

1. **按标题层级**：Markdown/HTML 的 `h1`～`h3`、Word 样式「标题 1/2」、PDF 书签（若可靠）作为硬边界；每个叶子小节单独成块或再细分。
2. **按条款编号**：如「第 3.2 条」「（一）（二）」作为边界，便于用户口头说「第三条」时仍能通过关键词 + 向量命中。
3. **列表与步骤**：有序列表尽量整块保留；若单条过长，按「一步一块」切，元数据中保留 `step_index`。
4. **表格**：表格不要与无关正文混在同一块；大表可按行组或「表头 + 前 N 行」成块，多块共享同一 `table_id`，检索命中任一块时可按需扩展相邻块（扩展策略可在本服务或入库时预链 `parent_chunk_id`）。

### 2.4 重叠（Overlap）与上下文窗

- **Overlap 的作用**：避免关键句落在两块边界上导致「哪块都不像答案」。  
- **典型取值**：中文场景下常见为块大小的 **10%～20%** 字符重叠，或固定 50～150 字；需与 **最大块长** 联调。  
- **与父子块（Parent-Child）的关系**：子块用于检索（小块向量），父块用于生成（大块全文）；addrag 若实现 parent-child，需在 metadata 中写入 `parent_id`，本服务检索到子块后可二次 fetch 父块内容（当前仓库若未实现，可作为后续能力）。

### 2.5 元数据（Metadata）设计建议

入库时除已有 `source`、`chunk_id` 外，可按业务扩展（Chroma `metadatas` 需可序列化标量，具体类型以 Chroma 版本为准）：

| 字段示例 | 用途 |
|----------|------|
| `doc_id` / `version` | 文档版本与回滚；多版本共存时用 `where` 只查当前版本 |
| `doc_type` | `policy` / `faq` / `sku` 等，便于 intent 与路由一致时过滤 |
| `section_path` | 如 `退换货/七天无理由`，提升可读性与 debug |
| `language` | 多语言库时分语言过滤 |
| `effective_from` / `effective_to` | 时效性规则，避免召回已废止条款（需本服务或中间层传「当前日期」参与过滤） |
| `acl` / `tenant` | 多租户时强制过滤，避免串库 |

**注意**：过滤字段要在入库时 **枚举值稳定**；随意改字符串会导致历史 `where` 失效。

### 2.6 是否存源文本：同库存储 vs 外部回表

向量库除了向量，还可以存 `documents`（chunk 文本）与 `metadatas`。是否把源文本放在向量库，取决于架构权衡，不是强制二选一。

- **同库存储（向量 + documents + metadata）**：检索后可直接拿 `snippet` 给 LLM，链路最短，开发成本低；你们当前 `query_chroma` 的返回方式就是这一类。  
- **外部回表（向量 + metadata，正文在关系库/对象存储）**：向量库只保留 `doc_id/chunk_id/version/tenant` 等键，召回后再去 MySQL/PostgreSQL 查正文。

推荐把以下键作为最小回表契约（无论是否同库存文本都建议保留）：

| 字段 | 作用 |
|------|------|
| `doc_id` | 关联原始文档主键 |
| `chunk_id` | 关联文档内分块序号或稳定分块 ID |
| `version` | 避免「向量索引版本」与「正文版本」错位 |
| `tenant` / `acl` | 多租户与权限过滤 |
| `source` | 人类可读定位（排障与审计） |

外部回表方案的关键注意点：

- **键稳定**：`doc_id/chunk_id` 必须在入库与查询链路中长期稳定。  
- **版本一致**：向量索引和关系库正文必须同版本发布/回滚。  
- **权限一致**：向量召回与正文查询都要做租户和 ACL 过滤。  
- **批量读取**：按 Top-K 一次批量回表，避免 N+1 查询放大延迟。  
- **可回溯**：日志里保留 `source/doc_id/chunk_id/version`，便于 badcase 复盘。

### 2.7 特殊内容：表格、代码、附件

- **表格**：纯文本拍平会丢列关系；可考虑 Markdown 表、或「键: 值」行化；复杂表可拆多行多块 + 共享 `table_id`。  
- **代码 / JSON**：保持 fence 或缩进，避免与说明文字混块；块内注明语言类型便于人工排查。  
- **图片 / PDF 扫描件**：若仅有 OCR，需在 addrag 做 **置信度与纠错**；噪声 OCR 会直接污染向量。

### 2.8 清洗与去重

- 去掉重复页眉页脚、站点导航、版权声明重复段，避免「千块一面」占满近邻。  
- 同一政策多 URL 爬取时做 **内容哈希去重** 或 canonical `source`。  
- 统一换行、全角半角、空白，减少无意义 token。

### 2.9 Embedding 与集合版本管理（强制一致）

- **换 embedding 模型** = 向量空间变化 → 必须 **全库重嵌入**，并建议使用 **新集合名或版本后缀**（如 `aicode_kb_v2`），避免与旧向量混查。  
- 本服务已通过集合 metadata 校验 `hnsw:space` 与 addrag 的 cosine 约定；**模型与维度**也应在运维文档中写明与 AICODE 对齐项。  
- 索引参数（Chroma HNSW `M`、`ef_construction` 等）若在 addrag 可调，变更后重建索引，并在发布说明中记录，便于排查「突然召回变差」。

### 2.10 与本服务消费方式的对齐

当前 `query_chroma` 将 `documents` 作为 `snippet` 返回；Summarizer 依赖这些片段。建议 addrag 侧：

- 控制单块 **不超过** 下游单次可承受的 token（并预留 rerank 多路合并空间）。  
- `snippet` 语义尽量 **自洽**（避免「……如下：」单独成块）。  
- `source` 人类可读且稳定，便于日志与客服复核。

## 3. Agent 编排

- **RAGTool** 仅返回证据、由 Summarizer 生成最终回复的分工合理；注意摘要展示与 `citations` 条数一致性问题（例如仅展示 top3 但 citations 含更大 `top_k` 时，应对齐「进入总结的 chunk 列表」）。  
- **意图与查询**：`rule` 路由传入 RAG 的文本是否包含足够规则名或场景词；可在路由层做槽位填充或规范化。  
- **多轮**：将会话中已确认的实体拼入检索 query，减少指代失败。

---

## 4. 工程与性能

- **HTTP / Chroma 客户端复用**：高频场景下对 Embeddings 的 `httpx` 客户端与 `PersistentClient` 做生命周期管理，降低延迟。  
- **缓存**：同会话、同规范化 query 的向量或检索结果短 TTL 缓存。  
- **可观测**：在 `rag_trace` 中记录 embedding 耗时、Chroma 耗时、是否过阈值、rerank 前后排名变化等，便于调参。

## 5. RAG 评估（建议流程与指标，文档级）

无评估时，难以判断分块、换模型、rerank、混合检索等改动的真实收益与回归风险。下面是一套**不依赖具体代码仓库**、可直接在表格或轻量脚本中落地的评估方案。

### 5.1 评估对象分层

| 层级 | 评什么 | 典型产出 |
|------|--------|----------|
| **检索层** | Top-K 是否包含「应引用的片段/文档」 | Recall@K、MRR、命中率、距离阈值通过率 |
| **生成层** | 最终回答是否覆盖要点、是否违背证据 | 要点 F1 / 人工打分、幻觉率 |
| **端到端** | 用户任务是否完成 | 任务成功率、转人工率、Badcase 列表 |

当前架构里 RAG 节点主要产出 **证据片段**（`citations` / `rag_trace`），Summarizer 再生成回复；建议至少固定跑 **检索层 Golden**，生成层用 **抽检 + 小集** 即可。

### 5.2 Golden Set（黄金集）怎么建

- **规模**：起步 50～200 条有代表性的真实或脱敏问句；覆盖高频意图、边界问法、多轮指代、专有名词。  
- **每条至少包含**：`query`（检索用问句）、**相关性判据**（任选或组合）：  
  - 期望命中的 `source` 子串或完整路径；  
  - 期望 `doc_id` / `chunk_id`（若入库稳定）；  
  - 期望正文中必须出现的短语（用于弱标注库）。  
- **可选**：标准答案要点列表（用于生成层「要点是否被回答」）。  
- **版本**：数据集标版本号；换 embedding 或重建索引后，用**同一套题**复跑对比，避免题目漂移。

### 5.3 检索层指标（与当前 Chroma 返回字段对齐）

- **Recall@K**：在 Top-K 条结果中，是否存在至少一条满足 Golden 中定义的相关性（与业务定义一致即可）。K 建议与线上 `RAG_TOP_K` 一致，另可报告 K=5/10 用于诊断「是否只是排得靠后」。  
- **MRR（Mean Reciprocal Rank）**：对每条 query，取**第一个**相关结果的名次 \(r\)，贡献 \(1/r\)；无相关结果记 0。适合比较「首位是否更准」。  
- **nDCG**（可选）：若标注为**多档相关**（如 0/1/2 分），可对排序质量更敏感；二值标注时 nDCG 与精心设计的 Recall/MRR 信息量部分重叠，非必须。  
- **空结果率 / 错误率**：检索 API 失败、集合为空、距离全劣于阈值的比例，单独统计，勿与「不相关」混为一谈。

### 5.4 生成层与忠实度（Faithfulness）

- **抽样**：从 Golden 或线上日志抽样，检查最终回复是否 **可被检索到的片段支持**（无捏造条款、无错误数字）。  
- **简单 rubric**：例如 1～3 分：完全有据 / 部分越界 / 明显幻觉；两名标注者一致性可 Kappa 粗看。  
- **与检索解耦**：若检索已错，生成层「对证据忠实」仍可能「对用户事实错误」——需在报告里区分 **检索错** vs **总结错**。

### 5.5 执行节奏与门禁

- **每次**：换 embedding、改分块策略、调 HNSW、上 rerank 前后，**同一 Golden 集**跑一遍检索指标并归档 JSON/表格。  
- **回归门禁**（示例）：Recall@K 下降超过约定阈值（如 2 个百分点）则阻塞发布，先查数据与索引版本。  
- **Badcase 闭环**：未命中条目记录「实际 Top-1 source / snippet 摘要」，回填为新一轮标注或分块优化输入。

### 5.6 与当前实现的对应关系（便于你对照日志）

- 检索结果中的 `source`、`chunk_id`、`snippet`、`distance`、`doc_id` 均可作为 Golden 匹配字段；`rag_trace.records` 便于按子任务追溯当次检索查询与命中列表。  
- 若线上仅向用户展示部分片段而 `citations` 更长，评估时应明确评的是 **「进入模型的上下文」** 还是 **「用户可见引用」**，两者 KPI 可分开报。

## 6. Embedding 模型与向量库选型（推荐）

以下为 **一般性推荐**，需与 **中文占比、部署形态（云 API / 本地 GPU）、成本、合规** 综合选择；**入库与查询必须使用同一模型与同一距离度量**，与本文第 2.8 节一致。

### 6.1 Embedding 模型（推荐思路）

| 类型 | 代表方向 | 适用场景 | 注意 |
|------|----------|----------|------|
| **云侧、多语种、省心** | OpenAI `text-embedding-3-small` / `3-large`，或同类兼容接口上的等价模型 | 中小知识库、与现有 OpenAI 协议栈一致（与本仓库 `rag_tool` 调用方式接近） | 成本按 token；大库全量重嵌入要算账单；维度与 `dimensions` 参数需与建库一致 |
| **中文语义为主** | 阿里通义等 **中文 embedding**、BAAI **bge-m3**（多语种）、智谱等厂商向量模型 | 客服、政策、站内文档以中文为主 | 确认 API 是否支持批量、速率限制；自建库时记录 **模型名 + 版本 + 向量维度** |
| **本地 / 私有化** | **bge-large-zh**、**m3e**、较小体量 **bge-small-zh** 等 | 数据不出域、已有 GPU | 需自建推理服务；运维与版本 pinning 更重要 |
| **重排序增强** | 检索仍用 bi-encoder；另购 **cross-encoder / rerank 专用模型**（云或本地） | Top-K 已召回但排序差 | rerank 与 embedding 是两条线，勿混为同一向量 |

**选型建议**：若以 **快速对齐 AICODE / OpenAI 生态** 为主，继续 **小模型 embedding + 适当 rerank** 往往性价比最高；若以 **中文长文档、术语密集** 为主，可对比评测 **中文向量化模型** 与 3-small 在同一 Golden 集上的 Recall@K，再决定是否迁移。**任何更换都须全量重嵌入并换新集合版本号**，避免旧向量混查。

### 6.2 向量库（推荐思路）

| 产品 | 特点 | 适用 |
|------|------|------|
| **Chroma（Persistent）** | 嵌入式、上手快、与 Python 生态结合紧；本仓库当前路径 | 单机、中小规模、开发与内测 |
| **Qdrant** | 过滤与混合检索能力较好，可 Docker / 云托管；REST 与客户端成熟 | 生产环境、需要 **payload 过滤**、多集合与扩展 |
| **Milvus / Zilliz Cloud** | 面向大规模与分布式；功能全 | 海量向量、团队有运维能力 |
| **pgvector（PostgreSQL）** | 向量与业务表同事务，JOIN 方便 | 已有强 PG 栈、向量规模中等 |
| **Pinecone / Weaviate 等** | 托管或集群化产品化程度高 | 希望少运维、接受 SaaS 或集群成本 |

**选型建议**：当前阶段 **保留 Chroma 作为默认路径** 合理；若出现 **高并发、强元数据过滤、多租户隔离** 或需 **官方水平扩展**，可评估迁移至 **Qdrant** 或 **pgvector**（取决于是否已深度使用 PG）。迁移时重点验证：**距离定义（cosine / ip / l2）与 embedding 归一化是否一致**、过滤语义、以及备份恢复策略。

---

## 7. 实施优先级（参考）

1. **数据与分块 + 元数据**（长期收益最大，与 addrag 联调）。  
2. **建立检索 Golden + Recall@K / MRR 基线**（本节与第 5 节）。  
3. **距离阈值 + 按文档去重 / 略大 top_k + rerank**。  
4. **查询改写**（按 badcase 类型选 HyDE 或规则扩展）。  
5. **混合检索**（在有关键词通道时）。  
6. **工程**：连接池、缓存、trace 完善；按需再评估向量库与 embedding 升级（第 6 节）。
