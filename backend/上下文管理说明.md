# 后端上下文管理说明

本文说明当前 `backend` 项目里上下文（context）如何创建、存储、更新和清理，便于你后续扩展复杂工作流。

## 1. 总体设计

系统使用 **SessionStore + GraphState + OrderContext** 的组合管理上下文：

- `SessionStore`：内存级会话仓库，按 `session_id` 管理状态。
- `GraphState`：单次消息在编排图（LangGraph）中的运行态，同时会回写到会话仓库作为长期态。
- `OrderContext`：订单子流程专用状态，跨多轮保留（例如补字段、确认执行、点击完成）。

核心原则：

- **短期执行态**：本轮图运行中通过 `state` 在节点间传递。
- **长期会话态**：每轮结束后落到 `SessionStore`，下一轮复用。
- **订单流程态独立**：订单相关字段和状态单独维护，不混到普通 query/rule 任务里。

## 2. 关键数据结构

定义位置：`app/core/state.py`

### 2.1 ConversationState（会话态）

- `session_id` / `user_id`
- `history`：用户与助手历史消息
- `last_intent` / `active_intent`：意图切换轨迹
- `turn_index`：轮次计数

用途：维护对话连续性和意图演进。

### 2.2 OrderContext（订单态）

- `operation`：`create/cancel/modify`
- `status`：`collecting_info/awaiting_pre_confirm/executed_waiting_click/closed/failed`
- `fields`：订单字段（地址、电话等）
- `items`：多商品清单（新增，用于整单下单）
- `order_link` / `failure_reason`

用途：支撑订单多轮交互，不因单轮结束丢失已收集信息。

### 2.3 GraphState（编排运行态）

包含以下关键域：

- 对话与流程：`conversation`、`order_workflow`
- 证据追踪：`rag_trace`、`sql_query_trace`
- 多任务执行：`sub_tasks`、`current_task_index`、`task_results`、`task_context`
- 可观测性：`observability`（request_id、节点耗时、日志、错误）
- 交互控制：`pending_actions`

用途：承载“一次用户输入”从拆分、调度、执行到总结的完整过程。

## 3. 生命周期（每轮消息）

主流程入口：`app/core/orchestrator.py -> process_message()`

1. **会话定位**
   - `SessionStore.ensure_session()` 保证 `session_id` 存在。
   - `get_or_create_graph_state()` 读取长期状态作为本轮初始态。

2. **初始化本轮 GraphState**
   - 复用历史态中的 `conversation/order_workflow/rag_trace/sql_query_trace/...`
   - 重置本轮执行相关字段（如 `sub_tasks/task_results/task_context`）。

3. **任务分析与调度**
   - `IntentRouterAgent.analyze()` 一次返回 `route + tasks`。
   - 编排器按 `depends_on` 选择就绪任务执行。

4. **节点执行并回填上下文**
   - `query/rule/order/complex` 节点写入 `raw`。
   - `collect_result` 把每个子任务结果写入：
     - `task_results`
     - `task_context[task_id]`
     - 并关联 `sql_query_trace/rag_trace` 的 task 级证据。

5. **总结与持久化**
   - `summarizer` 汇总输出。
   - `save_state` 将当前 `GraphState` 回写 `SessionStore`，并追加 `conversation.history`。

## 4. SessionStore 如何管理状态

定义位置：`app/core/session_store.py`

内部两个核心字典：

- `_graph_states: dict[session_id, GraphState]`
- `_orders: dict[session_id, OrderContext]`

关键方法：

- `get_or_create_graph_state()`：获取/初始化会话图状态。
- `get_or_create_order()`：获取/初始化订单上下文。
- `get_order()`：判断是否有活跃订单流程。
- `save_graph_state()`：本轮结束后持久化图状态。
- `append_history()`：写会话历史。
- `clear_order()`：订单闭环后清理订单态并重置 `order_workflow`。

> 当前实现为**进程内内存态**，服务重启会丢失状态。生产建议替换为 Redis/DB 持久化。

---

## 5. 多任务上下文（task_context）怎么用

`task_context` 是跨子任务传递数据的关键：

- 键：`task_id`
- 值：该任务的结构化上下文（状态、消息、sql、citations、outputs 等）

典型场景：

- 先查询“哪些商品降价”（query 任务）
- 再“把降价商品下单”（order 任务）
- order 任务通过 `depends_on` 读取前序任务 `outputs.proposed_order_items`

这样可以做到：

- 子任务间数据传递清晰
- 总结阶段可按任务聚合证据
- 调试时可按 task 追踪来源

---

## 6. 订单上下文的特殊管理

关键点在 `orchestrator._order_node` 与 `order_chain.py`：

- 复杂依赖场景下，系统会先生成拟下单清单。
- 现在会把拟定商品写入 `OrderContext.items`，并设置 `operation=create`。
- 若缺地址/电话，状态保持 `collecting_info`，用户补齐后进入 `awaiting_pre_confirm`。
- 用户确认后执行 `create_order`，携带整单 `items`，返回订单链接后进入 `executed_waiting_click`。

这保证了“先查降价 -> 再下单”不会在确认时丢上下文。

---

## 7. 追踪上下文：RAG 与 SQL

- `rag_trace.records[]`：按 `task_id` 记录检索查询与命中文档。
- `sql_query_trace.records[]`：按 `task_id` 记录 SQL、引用证据、订单明细。

设计目标：

- 避免多任务覆盖（每个任务独立 record）
- 为 Summarizer 提供可追溯证据
- 支持后续可观测性与调试面板

---

## 8. 当前边界与建议

当前边界：

- 仅内存存储，重启即丢失。
- 并发/多实例下没有分布式一致性保障。
- `GraphState` 与 `OrderContext` 部分字段存在冗余（可进一步收敛）。

建议演进：

- 将 `SessionStore` 抽象成接口，落地 Redis（短期）+ DB（长期）。
- 给 `task_context.outputs` 定义严格 schema（Pydantic/TypedDict）。
- 增加 `session_version` 与乐观锁，避免并发写覆盖。
- 为上下文增加 TTL 与归档策略。

---

## 9. 一句话总结

当前上下文管理是“**会话图状态 + 独立订单状态 + 任务级证据追踪**”三层结构：  
能满足多轮、多任务、依赖传递与可追溯，但持久化与并发一致性还需要进一步工程化。
