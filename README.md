# Agent + RAG 智能助手（游戏社区 · 玩家智能客服）

## 项目简介

**产品侧**：面向**游戏社交平台**普通玩家的 **Web 智能客服**——覆盖限时抢购、订单、发帖审核、好友与圈子、充值到账、举报与安全等常见咨询。玩家通过浏览器登录后进入对话页；**首次进入空会话**时展示可点击的**快捷问题**，一键填入输入框，减少打字成本。

**技术侧**：客服大脑由基于 Python 的「检索增强生成（RAG）+ 工具/多步推理 Agent」支撑：在可配置的 LLM 后端上，对平台规则、FAQ、活动说明等文档进行向量化检索，将检索片段注入对话上下文，并支持工具调用（如订单查询占位、当前时间、再次检索等）。**双 LLM**（云端 OpenAI 兼容 API 与本地 Ollama 等）通过配置切换。

**典型场景（MVP 边界）**

- 玩家：在 `frontend` 聊天框多轮提问，获得带知识库依据的答复（后端落地后）；当前无后端时前端为演示回复。
- 运营/部署方：维护客服知识库（PDF / TXT / Markdown），RAG 检索后回答并展示引用片段。
- 任务型对话：Agent 在检索基础上多步推理并调用工具（MVP 演示级即可）。

MVP 聚焦「单部署、配置驱动」；不承诺多租户 SaaS 与复杂权限。**玩家侧登录**当前为演示实现（本地存储），正式环境需替换为平台统一鉴权。

## 前端界面（仓库内已实现）

路径：**[frontend/](./frontend/)**（Vite + React + TypeScript + React Router）。

| 能力 | 说明 |
|------|------|
| 登录页 | 演示：任意昵称 + 至少 4 位密码进入；文案定位为「游戏社区 · 智能客服」。 |
| 对话页 | 多轮消息、本地按用户持久化、清空会话、退出登录。 |
| 快捷问题 | 空会话时展示「玩家常问」（抢购、订单、发帖、社交、充值、举报等），点击填入输入框。 |
| 对接后端 | 可选环境变量 `VITE_API_BASE_URL`；未设置时使用内置演示文案。配置后请求 `POST {base}/api/chat`，请求体与响应约定见 `frontend/src/lib/chatApi.ts` 注释。 |

**本地运行前端**

```bash
cd frontend
npm install
npm run dev
```

复制 `frontend/.env.example` 为 `frontend/.env` 并按需填写 `VITE_API_BASE_URL`。对接自建 API 时注意 **CORS** 与 **HTTPS/混合内容** 策略。

## 后端 Phase 0（云端 OpenAI）配置

在仓库根目录创建 `.env`（用于后端 FastAPI + CLI 读取），只需设置最少项即可：

- `LLM_PROVIDER=openai`
- `OPENAI_API_KEY=...`
- `OPENAI_BASE_URL`（可选，默认 `https://api.openai.com/v1`）
- `OPENAI_MODEL`（可选，默认 `gpt-3.5-turbo`）
- `OPENAI_TEMPERATURE`（可选，默认 `0.2`）

启动后端：

```bash
uvicorn assistant.api.app:app --host 0.0.0.0 --port 8000
```

当前后端代码结构已迁移为 `src/` 同级模块，核心目录如下：

```text
src/
├── settings.py
├── aicode_types.py
├── llm/
├── embeddings/
├── rag/
├── addrag/
└── assistant/        # 兼容入口壳（API/CLI），逐步收敛
```

注意：
- 请使用 `aicode_types.py` 作为类型定义模块。
- 不要在 `src/` 根目录创建 `types.py`，该文件名会与 Python 标准库 `types` 冲突并导致服务无法启动。

独立向量写入（与后端查询解耦）：

```bash
python -m addrag ingest-md
```

- 默认读取目录：`RAG_SEED_DIR`（未配置时为 `data/seed`）
- 严格只读取 `.md/.markdown`
- 采用幂等 `upsert`，重复执行会覆盖同 id 分块

## 术语

| 术语 | 说明 |
|------|------|
| **RAG** | Retrieval-Augmented Generation：先检索相关文档片段，再让模型基于片段生成回答。 |
| **Embedding** | 将文本映射为稠密向量，用于语义相似度检索。 |
| **Vector Store** | 存储与查询向量及元数据的组件（如 Chroma、FAISS、pgvector）。 |
| **Agent** | 在对话中按策略多步执行：推理、选工具、执行、再推理，直至给出最终答复或达到限制。 |
| **Tool** | 可被 Agent 调用的外部能力（函数/API），返回结构化或文本结果供模型继续推理。 |
| **Session** | 一次连续对话的上下文与会话状态（消息历史、可选会话 ID）。 |

## 用户与场景

**目标用户**

- **玩家**：通过 Web 智能客服自助查询规则与常见问题（与前端文案、快捷问题一致）。
- **开发者 / 小团队**：自建后端与知识库，自备 API 密钥或本地模型环境。

**核心用例（优先级）**

| 优先级 | 用例 |
|--------|------|
| **P0** | 玩家使用 `frontend`：登录 → 空会话快捷问题或自由输入 → 多轮对话（后端接入后为真实模型+RAG）。 |
| **P0** | 导入文档 → 向量化入库 → 针对知识库提问 → 回答中展示引用片段。 |
| **P0** | 模型后端可通过配置在「云端 OpenAI 兼容 API」与「本地 Ollama（或同类）」之间切换。 |
| **P0** | Agent：至少支持「检索后再答」链路；内置简单工具或等价演示。 |
| **P1** | 流式输出（首 token 起逐步展示）；前端可在后续对接 SSE/WebSocket。 |
| **P1** | 请求级日志、Token 用量占位、简单 tracing 扩展点。 |

## 功能需求

### 玩家 Web 客服界面（`frontend`）

- 登录入口与客服对话主流程；品牌与文案与「游戏社区、抢购、发帖、社交」等业务一致（可替换为实际平台名）。
- 空会话下的快捷问题列表，点击写入输入框（可编辑后再发送）。
- 与后端约定的聊天接口（见 `frontend/src/lib/chatApi.ts`）；错误时在界面可见提示。

### 对话（后端）

- 多轮会话与上下文管理（含历史截断/摘要策略占位，MVP 可采用长度截断）。
- **流式输出：P1**（MVP 可先非流式，计划中 Phase 3 补齐）。

### RAG

- 文档上传或指定路径导入；解析 **PDF、TXT、Markdown**（MVP 格式范围）。知识库内容建议包含：活动规则、社区规范、支付与售后说明等，以支撑客服场景。
- 文本分块（可配置块大小与重叠）、Embedding、写入向量库。
- 检索：Top-K 召回；**重排序为占位/后续 backlog**。
- 回答中展示引用片段（来源标识、可定位到块或文件名）。

### Agent

- 在检索结果基础上推理；工具调用框架可插拔。
- MVP 工具示例：计算器、当前时间、**检索**作为可被 Agent 显式调用的工具，以及面向游戏业务的固定工具，例如：  
  - `get_user_orders`：基于**本地 MySQL** 数据库按玩家账号/订单号查询最近订单状态；  
  - `get_user_coupons`：基于 MySQL 查询玩家当前可用优惠券与使用规则；  
  - `handoff_to_human`：对当前会话做 summary，并将玩家信息 + summary 通过后端推送到人工客服/工单系统。  
  所有工具均通过单一的 `/api/chat` 接口间接调用，由后端在 Agent 流程中决定何时调用哪个工具。
- 失败重试策略、最大步数/超时上限，避免死循环。

### 模型后端（双 LLM）

- **统一抽象**：同一套对话/补全接口，通过配置切换实现。
- **云端**：OpenAI 兼容 HTTP API（`base_url` + `api_key` 等）。
- **本地**：Ollama 或同类本地推理服务（兼容 OpenAI 风格端点更佳）。
- **配置约定**：支持环境变量与配置文件（如 `.env` + YAML/TOML）；**文档中只写变量名与含义，不出现真实密钥**。密钥不入版本库。

### 可观测性（P1）

- 请求级日志（可脱敏）。
- Token 用量统计占位（便于后续计费/限额）。
- Tracing 钩子说明（可选对接 OpenTelemetry 等，MVP 可不实现仅留接口）。

## 非功能需求

**性能（定性）**

- 首 token 延迟：在本地小模型与合理网络下，应尽量接近所选模型与链路的正常水平；不规定绝对毫秒值。
- 检索耗时：单库 Top-K 应在交互可接受范围内（通常亚秒到数秒级，视库规模与硬件而定）。
- 玩家侧页面首屏与发送响应：保持常见网络下可接受的交互延迟（前端以异步请求与状态提示为主）。

**安全**

- API 密钥仅通过环境变量或本地配置文件提供，**不入库、不写入日志明文**。
- 上传文件类型白名单与大小上限；服务端路径与存储路径规范化，**防止路径遍历**。
- 正式环境：**玩家鉴权**应对接平台账号体系，替代演示用本地登录；客服对话与知识库访问需符合平台隐私与审计要求。
- 对用户上传内容不做默认外发；合规由部署方与数据提供方负责。

**可维护性**

- 后端模块化目录（RAG、Agent、LLM、ingest 等边界清晰）；前端组件与常量（如快捷问题）独立可改。
- Python 类型标注；核心逻辑具备最小单元/集成测试（见 `plan.md`）。

**合规**

- 用户自备数据与第三方 API 须自行满足所在地法律与合同要求；本项目文档不提供法律意见。

## 验收标准（与 `plan.md` 里程碑对齐）

1. **前端**：可运行 `frontend`，完成演示登录；空会话展示快捷问题并可填入输入框；发送消息有反馈（演示文案或真实 API）。
2. **后端 API（接入后）**：`POST /api/chat` 与 `frontend/src/lib/chatApi.ts` 约定一致（或可配置路径），跨域（CORS）在开发/部署下可用。
3. **RAG**：可导入至少一种 MVP 格式文档，提问后返回答案，且答案附带可识别的引用片段（文件/块级均可）。
4. **双 LLM**：通过配置切换云端 OpenAI 兼容端点与本地 Ollama（或文档约定的同类），无需改业务代码路径（仅改配置）。
5. **Agent**：演示至少一步工具调用 **或** 一步明确的「检索后再答」链路（与 plan 中 Phase 2 检查清单一致）。
6. **可运行**：README/plan 中提供后端最小运行说明（Phase 0/3 交付物），新环境可完成依赖安装与一次端到端提问（经 CLI 或经前端）。

## Phase 2 运行边界（Agent vs Phase 1）

- 前端继续只调用 `POST /api/chat`，不直接触达任何工具接口。
- `AGENT_ENABLED=false`：走 Phase 1 路径（自动 RAG 注入 system prompt 后再回答）。
- `AGENT_ENABLED=true`：走 LangGraph Agent 路径（由 Agent 决定是否调用 `retrieve_knowledge`、`get_user_orders`、`get_user_coupons`、`handoff_to_human`）。
- 无论哪条路径，接口都返回 `{ reply, citations }`；其中 citations 在 Agent 模式下仅来自本轮工具检索结果。

后端测试需要 **Python 3.11+**，并安装开发依赖：`pip install -e ".[dev]"`，再在项目根目录执行 `pytest`。

## Phase 2 快速验收示例

- 工具调用：`帮我计算 (29+11)*2`（可触发 `calculator`）。
- 检索后回答：`根据知识库说明退款流程`（可触发 `retrieve_knowledge`）。
- 订单/优惠券：`查询 player_001 的订单和优惠券`（可触发 Mock 业务工具）。
- 转人工：`这个问题你处理不了，转人工并总结一下`（可触发 `handoff_to_human`，生成 `MOCK-` 工单号）。
- 安全停止：将 `AGENT_MAX_STEPS` 调小后提复杂多跳问题，应返回超限/超时的可读提示，而不是 500。
