# 多 Agent 客服系统需求文档

## 1. 项目背景
构建一个面向文具业务场景的多 Agent 协作客服后端系统。系统接收用户消息后，先完成意图识别，再分发到对应 Agent 执行，最后由 Summarizer Agent 汇总成最终回复。

## 2. 总体目标
- 实现基于图（Graph）的多 Agent 编排链路：`IntentRouter -> (SearchAgent | RAGAgent | OrderAgent) -> SummarizerAgent`
- 支持订单流程强管控（基于 LangChain chain）
- 支持本地大模型与云端大模型切换
- RAG 基于本地向量库检索（`D:\AICODE\data\chroma`）
- 前端改造仅在副本中进行，不改原前端

## 2.1 图编排流程（多问题顺序执行）
```mermaid
flowchart TD
    A[用户输入] --> B[load_state<br/>加载/初始化会话态]
    B --> C[decompose<br/>拆分子任务 sub_tasks]
    C --> D[dispatch<br/>按 current_task_index 选择当前任务]

    D -->|query| E[query_agent<br/>SearchAgent]
    D -->|rule| F[rule_agent<br/>RAGAgent]
    D -->|order| G[order_agent<br/>OrderAgent]
    D -->|unknown| H[unknown_agent]
    D -->|handoff| I[handoff_node]
    D -->|safe_response| J[safe_response]
    D -->|all_done| K[summarize<br/>SummarizerAgent]

    E --> L[collect_result<br/>记录 task_results / pending_actions]
    F --> L
    G --> L
    H --> L
    I --> L
    J --> L

    L --> D
    K --> M[save_state<br/>保存会话与图状态]
    M --> N[END]

    O[/orders/confirm] --> P[order_confirm]
    Q[/orders/finalize] --> R[order_finalize]
    P --> M
    R --> M
```

流程约束：
- 图采用任务循环执行：`dispatch -> agent -> collect_result -> dispatch`
- 当 `current_task_index >= len(sub_tasks)` 时，路由到 `all_done`，进入 `summarize`
- 订单任务进入 `awaiting_pre_confirm` 后写入 `pending_actions`，必须用户确认后再执行
- 最终统一经过 `save_state` 持久化，保证一次请求任务闭环

## 3. 业务需求

### 3.1 意图识别
用户消息需识别为以下类型之一：
- `query`：查询类问题
- `rule`：规则类问题
- `order`：订单类问题（强控制）
- `unknown`：未识别

实现约束：
- 意图识别由大模型主导，不使用硬编码规则主导分流
- 建议输出结构化结果（intent/confidence/reason），并支持置信度阈值控制

### 3.2 Search Agent（查询类）
- 根据查询问题调用搜索工具（当前可为 mock）
- 若命中结果，返回可读摘要及引用信息
- 若无命中，返回明确错误提示（如补充关键词）

### 3.3 RAG Agent（规则类）
- 基于 `D:\AICODE\data\chroma` 执行检索
- 输出规则依据与引用片段
- 若无命中或检索异常，必须返回可解释错误

### 3.4 Order Agent（订单类，强控制）
订单操作包含：
- 下单
- 退单
- 修改订单信息

订单流程必须遵循：
- 收集必要信息
- 用户执行前确认
- 执行订单操作（默认不调用支付 API、不扣余额）
- 返回订单/退款/修改结果链接
- 用户点击确认后流程结束

失败要求：
- 任一步骤失败必须反馈原因

实现要求：
- 使用 LangChain chain 实现强管控

### 3.5 Summarizer Agent
- 统一对各子 Agent 输出进行整理
- 生成最终回复文本给用户
- 保留必要动作提示（如继续补充信息、点击链接确认）

## 4. 大模型能力需求
- Agent 基于大模型能力驱动
- 支持两种模式切换：
  - 本地模型（如 Ollama）
  - 云端模型（如 OpenAI 兼容接口）
- 通过配置项控制模式与模型参数

## 5. 接口需求（/api/v1）
- `GET /api/v1/health`：健康检查
- `POST /api/v1/chat/message`：消息入口，触发意图识别和 Agent 编排
- `POST /api/v1/orders/confirm`：订单执行前确认入口
- `POST /api/v1/orders/finalize`：用户点击订单链接后的收尾确认

说明：
- 已废弃并移除 `orders/execute`，执行动作应由 chain 在确认后内部推进。

## 6. 前端协同需求
- 原前端路径：`D:\AICODE\frontend`
- 仅改造副本（位于 `D:\Agents`），对接 `/api/v1` 新协议
- 订单场景需支持：
  - 展示订单结果链接
  - 用户点击后触发 finalize 完成流程

## 7. 非功能需求
- 错误处理明确、可读
- 接口字段结构稳定，便于前端消费
- 支持后续将 mock 工具替换为真实服务

## 8. 范围边界
- 本阶段允许工具层返回 mock 数据
- 本阶段不接入真实支付扣款流程
- 本阶段不要求实现完整会话持久化数据库
